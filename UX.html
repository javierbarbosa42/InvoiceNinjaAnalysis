UX

Let's analyze the UX. In this case, we will use the night mode. Given that it wasn't exposed on the UI side of the application. For this process, we will show gifs (or gifs) in small format, exposing the UX fallacies that are found throughout the use of the application. Some of the errors that are showed were provoked and then replicated, others were simply found and tried to be replicated.

The fallacies that could be replicated were recorded. But fallacies don't mean just "errors/bugs".

They also mean things that can affect the user experience regardless if they are bugs. Let's take a look at the first one for example.

This captures were done on a Samsung Galaxy s6


In this first gif (or gif). We see the app when it is just started. Right of the bat we can notice the long wait time required for the application to deploy. This is after opening it and closing it multiple times. This long time affects the user experience; with a long wait for the application to be started

As we see in the gif above, the wait time for the application to start is unusually long. But there is something else to notice in this small gif.

Notice that right when you start the application you are placed on the last open feature. This happens with this and with every other feature, including the settings feature (there is a special case with the settings which will be showed later). This means that there is no home page, or landing page. When the user starts the application (even when closing the application), he is placed on the last used feature, there should be a landing page (a page like the dashboard), that is easily recognizile as the first thing the user sees. This to provide structure to the application, it gives the user something to familiarize himself, a home page.

That is not all though. It's not noticeable, but when the back button is clicked there is a huge wait that takes you to the dashboard. Again, it is not noticeable because we can't show the back button, but believe s when we say that the first thing we do is click the back button right after landing on the vendor page


Let's look at the second gif

In this second gif we can notice several different fallacies.

  • First, we see that when in the add a new vendor page there are a lot of spaces to complete. Clicking on one of this spaces opens up the keyboard. That's perfectly fine. But when clicking another tab inside this page we can notice that the keyboard does not close. It keeps on even after there would be nowhere to write because there is no writing spaces clicked or active at the moment
  • Second, we see that when the user is in the add a new vendor page and he exits the application, then goes back to the application, he is placed again in the add vendor page. This has a lot to do with the error exposed on the fist gif. The user is not placed on a home-screen. He is placed on his last active screen.
  • Third, on that same section of the gif. We can notice that when the back button is pushed, and the user goes back to the vendors page. The filter is still on. The user is not shown his vendors, as he should, but rather the filter on with the filter that was written a while ago, and that stuck even after closing the app

In this third gif, we were able to replicate an error when the application launches. As it can be seen below, we see that sometimes, when the application launces, the user is taken directly to the last active page (we have explained this before so let's move on), but besides that, the user is taken to the filter page but the real fallacy is that the user is placed directly as if he wanted to filter out content, full keyboard on. As if the user wanted to filter out something right when he starts the application


In this fourth gif we can see some other things, we will do our best to explain what is going on and what we can analyze from this:

First, we are placed in the Dashboard screen, this is from where the gif starts. From here what the user (we) does, is to click feature after feature, just going screen after screen, imagine a scenario where the user does a couple of things in a screen, then goes to another screen and does a couple of things, then another and so on. This is all good and dandy. Works perfectly fine.

Then imagine a scenario where the user does something on a screen, goes to another screen but quickly remembers something he had to do or see on the previous screen. What is the natural reaction? click the back button of the phone (not visible given the phone model but is the default back button). But what happens after the user clicks the back button? Look at what happens after the invoice screen.

The user goes to the Dashboard screen. He is not placed on the previous screen. He is placed on the Dashboard screen.

But that is not all. After invoice -> Dashboard, the user goes to quotes, and hit's the back button again. What happens? The user goes to the Dashboard screen, as explained earlier. But then in this screen the users hits the back button again, what happens in this case? The user is exited out of the application. Not only he is exited out of the application, but if we look closely we notice that there is no confirmation or any type of notification telling the user that hitting the back button will exit the application. There is no message telling the user to hit the back button twice to exit, or anything similar, he is just expelled out of the app.

Now in all fairness, going to the Dashboard when hitting the back button is probably nothing to worry about, in fact, it maybe was probably intended this way. If the user accessed all the screens one after another going back would take forever. This is a logic that could be user to have this as intended.
