Bugs & EC

Let's see how the application behaves when regars to eventual connectivity. But first, apply a literary technique called "frame story".

A frame story is a story that begins in the ending. So that's what we will do, we will begin by the ending. We will begin conclusion.

In conclusion there is basically no eventual connectivity techniques used in the applications. As we will soon see, there is little to no alert or notification when it comes to connection. We will see that the application does not notify in any way to the user that he/she has no connection.

When perfomirming an action or task, the user is not told that the task can't be completed. It just isn't completed.


Let's look at the first example of eventual connectivity:

As can be seen in this example above, when the user looses connection, whether that he/she lost wifi and data connection by purpouse or by accident. The user recieves no alert or notification telling him coonnection has been lost.

Adding to that, we can see that even changing the screen does not give any type of alert.


Let's get a little more in detail about the eventual connectivity on the application:

Here, we can see a very clear example of a bad use of eventual connectivity. In this specific case, there is a problem in regars with a progress that get's stucked, with no way to move forward. In other words, stuck progress notification.

As we said before, the user is not notified that the application has lost connection. But in this specific case we see a case of stuck progress notification. In which the user isn't notified that the connection has been lost and besides that, when we try to load something, we can just barely see the progress bar move and then nothing happening.

Stuck progress notification is a termn extracted from the book. In there it says that this case describes when a progress notification gets locked (not necessarily blocking the app) because of a connection problem.

Now we understand that there is a small differece between this definition and what is actually happening on screen. This definition talks about a progress bar that get's stuck loading. Therefore it never finishes, however in this case we can se that the progress bar loads for a second and then stops. Nevertheless, this definition is the closest one to what is actually happening on screen. This because we can just barely see the progress notification and nothing happens afterwards and also because we can see that the application itself is not locked. We can continue to use the application (technically), although we are not goint to be able to do much with it.
