Perfor-
mance

For the performance analysis of the app we took every feature of the app and gave it heavy use for a period of time to see how it responded on the phone. To achieve this, we used the android studio profiler on the android-side part of the app. This gave us a tool to see a graphic representation of how the app was performing on different aspects as CPU usage, memory usage, network usage and energy. After running these tests, we found that every feature behaved similarly. The reason for this at the conclusions section.


Dashboard



The abobe graph is showed as an overview of the performance. For the test, this view was reloaded and played with many times as it can be seen in the interaction section (represented as the red dots on top of the image). We tried all possible interactions of this view to test how well it was managed. We will now dive on detail on the different aspects the profiler shows.


CPU



Even though the graph shows moments where there are more than one thread running at the same time, this are used for little cases and most of them are used in Future calls and on async methods. This might show how the usage of multithreading is being used, but by reading the docs we know dart is a single threaded language, although there is Isolates which can serve as new threads in execution. We will discuss threading in the Multithreading section.


Memory



As shown above, the memory for this feature never surpasses the 256MB limit. Most of this memory is used is for the app heap to run code. If we divide the memory used by packages, the shallow size for the different packages using memory are the java package for the use of primitive values and data structures.


Network



For the network analysis, because of the nature of the feature, the network usage is as little as the user wants to reload the data in the view. For this reason, there is almost no network usage between the requests for reload.


Energy



There is a positive correlation between the network usage and the energy peaks on the above graph. Although there is peaks in the energy consumption, this consumption is always categorized as light.



Clients



For the clients overview the above graph shows a similar behavior as the dashboard feature. Low use of network and energy outstand in this overview. Similar usage of memory and very low usage of CPU except for the launching of the app (beginning). It important to note how in the activity representation of the app (green line below the interactions in red dots), due to one of the features of the app that let send emails to their clients, at the moment of tapping that option, the app opens up the default emailing app of the phone so Invoice Ninja is put to sleep.


CPU



The CPU usage for this feature is similar to the Dashboard, giving an average use of less than 20% inside the app.


Memory



Bear in mind that there is a gap at the start of the trace that is recording memory usage outside the app. Once again, the biggest use of memory is by native libraries, followed by code and graphics. There is another category that sums the rest of usages of memory called “Others” that is also important.


Network



The network is used in this feature is used to retrieve data from the server on client contacts. That is why the graph shows peaks at the beginning (fetching) and at the end of the profiling, where new contacts were added.


Energy



The energy profiling of this feature shows how the energy use is low for this feature.
