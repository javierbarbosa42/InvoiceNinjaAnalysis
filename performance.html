Performance

Dynamic Profiling



For the dynamic analysis of the app we took every feature of the app and gave it heavy use for a period of time to see how it responded on the phone. To achieve this, we used the android studio profiler on the android-side part of the app. This gave us a tool to see a graphic representation of how the app was performing on different aspects as CPU usage, memory usage, network usage and energy. After running these tests, we found that every feature behaved similarly. The reason for this at the conclusions section.


Dashboard



The abobe graph is showed as an overview of the performance. For the test, this view was reloaded and played with many times as it can be seen in the interaction section (represented as the red dots on top of the image). We tried all possible interactions of this view to test how well it was managed. We will now dive on detail on the different aspects the profiler shows.


CPU



Even though the graph shows moments where there are more than one thread running at the same time, this are used for little cases and most of them are used in Future calls and on async methods. This might show how the usage of multithreading is being used, but by reading the docs we know dart is a single threaded language, although there is Isolates which can serve as new threads in execution. We will discuss threading in the Multithreading section.


Memory



As shown above, the memory for this feature never surpasses the 256MB limit. Most of this memory is used is for the app heap to run code. If we divide the memory used by packages, the shallow size for the different packages using memory are the java package for the use of primitive values and data structures.


Network



For the network analysis, because of the nature of the feature, the network usage is as little as the user wants to reload the data in the view. For this reason, there is almost no network usage between the requests for reload.


Energy



There is a positive correlation between the network usage and the energy peaks on the above graph. Although there are peaks in the energy consumption, this consumption is always categorized as light.

It is normal for the memory to rise a bit once the image is taken, but with our tests we found out that the images that we uploaded are in average 5MB of size but the rise on memory shown in the profiling is more than 5MB, which we can conclude to be a memory leak. Reading in depth about memory leaks in flutter, we see there is issues on the flutter github repo regarding memory leaks, there is many reasons why this could be happening, but there is a probable cause; the state it's not being disposed properly after alterning from apps.

Static Profiling



The static profiling tool used was the dart analyzer that comes with the flutter plugin for android studio. We found out that this analyzer is very customizable but also strict out of the box.
We were surprised to find out that running this static analyzer found no code errors on the Invoice Ninja project. This probably has to do with the fact that Hillel Coren, CEO and main developer of the flutter app is actually a GDE (google developer expert) for flutter. We were still hesitant whether to trust this analyzer, so we tested it on our own project (transport app) and got a bittersweet feeling finding out that there was almost 70 warnings and bugs.
In conclusion, the Invoice Ninja project doesn’t have even a single error or warning.





Multithreading



Dart executes one operation at a time, one after the other meaning that as long as one operation is executing, it cannot be interrupted by any other Dart code. In other words, Dart in which flutter is based, is a single threaded language.


One might think that the usage of an object called Future or the async statement on a method is a way of parallel processing, but it actually is not. There is a single internal and invisible process called the "Event Loop" (that has an event queue to keep track of the code to be executed) in which a Future or async method interacts by pushing code into the event queue. There is many usages of both the Future and async methods in the project but these are not different threads ("Isolate" in flutter lingo) running.


(lib\data\repositories\client_repository.dart)


Instead, an Isolate is the implementation of the notion of Thread but there is a major difference, Isolates do not share memory between them. Invoice Ninja does not make use of this type of multithreading technique.
